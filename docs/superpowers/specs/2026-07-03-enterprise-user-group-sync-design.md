# Enterprise User Group Sync Design

## Goal

Add a guarded write tool for Stack Overflow Enterprise that creates and updates user groups from an uploaded user export CSV.

The first preset is narrow by design:

- CSV source: user export upload.
- Grouping column: `Senior Manager`.
- Member lookup column: `Email`.
- Stack user resolution: Enterprise API lookup by email.
- Default write behavior: add-only.
- Optional write behavior: exact sync.
- Only configurable field: user group name template.

This keeps the workflow reusable enough for future naming changes without turning the first write tool into a general workflow builder.

## Primary User Workflow

1. User opens the Enterprise write tools area.
2. User selects the User Group Sync tool.
3. User uploads a CSV with the expected user export columns.
4. User reviews or edits the group name template.
5. User leaves sync mode as add-only, or explicitly enables exact sync.
6. User runs a preview.
7. The app resolves CSV emails to Stack user IDs, groups users by senior manager, checks existing Enterprise user groups, and shows planned changes.
8. User confirms the preview.
9. The app applies the selected changes through the Enterprise user group endpoints.
10. The app shows an audit-friendly result summary with created groups, updated groups, added members, removed members, skipped rows, and API failures.

## CSV Contract

The uploaded CSV must include these headers:

- `Director`
- `Senior Manager`
- `User Group Member`
- `First Name`
- `Last Name`
- `Colleague ID`
- `Email`
- `Job Title`

The first preset uses only `Senior Manager` and `Email` for write planning.

Rows are ignored when:

- `Senior Manager` is blank.
- `Email` is blank.
- The email cannot be resolved to a Stack Enterprise user ID.

Ignored rows appear in the preview and final summary with a reason.

## Group Name Template

The default template is:

```text
{Senior Manager} VRM
```

The template is the only dynamic setting in the first implementation. It may contain the `Senior Manager` placeholder and static text. For example:

```text
{Senior Manager} VRM
VRM - {Senior Manager}
{Senior Manager} Team
```

The app trims generated names and rejects blank names. If two distinct senior manager values produce the same group name after trimming, the preview blocks apply until the conflict is resolved by changing the template or the CSV input.

## Enterprise API Flow

The tool uses the Enterprise v3 API described in the provided OpenAPI spec.

Read operations:

- Resolve each member email with `GET /users/by-email/{email}`, URL-encoding the email value.
- Load existing groups with `GET /user-groups`.

Write operations:

- Create a missing group with `POST /user-groups`, including the generated name and resolved `userIds`.
- Add members to an existing group with `POST /user-groups/{userGroupId}/members`.
- In exact sync mode only, remove extra members with `DELETE /user-groups/{userGroupId}/members/{userId}`.

Team-scoped equivalents under `/teams/{team}/...` can be used later if the app introduces team-specific execution. The first implementation targets Enterprise main-site endpoints.

The access token must include `write_access`.

## Sync Modes

### Add-Only

Add-only is the default and safest mode.

For each generated group:

- Create the group if it does not exist.
- Add resolved Stack users who are not already group members.
- Leave all existing group members untouched.

### Exact Sync

Exact sync is opt-in through a checkbox and requires preview confirmation.

For each generated group:

- Create the group if it does not exist.
- Add resolved Stack users who are missing.
- Remove existing members who are not present in the latest CSV-derived membership.

Exact sync only operates on groups generated by the current template and CSV input. It must not clean up unrelated groups.

## Preview Model

Preview is required before any write request.

The preview shows:

- Groups to create.
- Existing groups to update.
- Members to add.
- Members to remove when exact sync is enabled.
- Rows skipped because of missing manager, missing email, unresolved email, duplicate email, or naming conflicts.
- Credential or permission errors encountered during API lookups.

Apply is disabled when the preview has blocking errors such as duplicate generated group names or failed existing group lookup.

## Error Handling

CSV errors are local validation errors and should not call the write endpoints.

Email resolution errors are row-level failures. The preview can continue resolving the rest of the file and report unresolved users.

Group lookup failures are blocking because the tool cannot safely decide whether to create or update groups.

Write failures are operation-level failures. The app should continue where safe, record each failed API call, and present a final summary that is clear enough to retry manually or rerun after fixing permissions/data.

Exact sync removals should be listed separately from additions in both preview and final summary.

## UI Design

Add a guarded write-tools area separate from read-only reports.

The User Group Sync screen contains:

- CSV upload control.
- Group name template input, defaulting to `{Senior Manager} VRM`.
- Sync mode checkbox: `Exact sync`.
- Preview button.
- Preview tables for groups, member additions, optional removals, and skipped rows.
- Apply button enabled only after a successful preview.
- Final run summary.

The exact sync checkbox should be visually grouped with a short warning that removals can occur when it is enabled.

Credentials stay session-only, consistent with the reporting MVP.

## Testing

Unit tests:

- CSV parser accepts the expected headers.
- Planner groups rows by `Senior Manager`.
- Group name template renders `{Senior Manager} VRM`.
- Blank managers and emails are skipped.
- Duplicate generated group names block apply.
- Add-only plans only creates and additions.
- Exact sync plans creates, additions, and removals.
- Email resolution failures become row-level skipped results.

API client tests:

- `GET /users/by-email/{email}` returns a user ID.
- `GET /user-groups` reads existing memberships.
- `POST /user-groups` sends the generated group name and user IDs.
- `POST /user-groups/{userGroupId}/members` sends member IDs.
- `DELETE /user-groups/{userGroupId}/members/{userId}` is only used in exact sync mode.

UI tests:

- Uploading a valid CSV enables preview.
- Preview shows planned group creation and member additions.
- Add-only mode does not show removals.
- Exact sync mode shows removals and requires confirmation.
- Apply is disabled when preview has blocking errors.

## Non-Goals

- A fully configurable column-mapping builder.
- Arbitrary group-by rules.
- SCIM user activation, deactivation, or deletion.
- API import for questions, answers, or articles.
- Team-scoped Enterprise writes.
- Persistent storage of uploaded CSVs, credentials, previews, or write results.

## Future Expansion

Future presets can reuse the same preview/apply engine if they still follow a CSV-to-user-group sync pattern. The first likely extension is another group name template, not a broader mapping model.
