# Stack API Utilities Reporting MVP Design

Date: 2026-07-03
Status: Design approved in discussion; written spec pending user review

## Summary

Build a static browser app that consolidates the browser-ready, read-only SO4T Python reporting utilities into one Stacks-based frontend. The MVP runs entirely in the user's browser, with session-only credentials and session-only report data. Users can run reports live against Stack Overflow for Teams / Stack Internal APIs, or upload existing CSV/JSON outputs generated by the current scripts.

The first release focuses on reporting. Write actions, SCIM mutations, and Selenium/admin-page scraping are excluded from browser execution and reserved for later phases.

## Goals

- Provide one frontend app for all browser-ready read-only reporting scripts.
- Run entirely in the browser with no backend in the MVP.
- Support both live API runs and uploaded existing CSV/JSON outputs.
- Support Stack Overflow for Teams Basic/Business and Stack Internal Enterprise where each report permits it.
- Use Stack Overflow's Stacks design system as the visual foundation.
- Give each report a dashboard plus raw table inspection with search, filtering, sorting, column visibility, and export.
- Support running one report at a time and running selected reports as a batch queue.
- Reuse shared API datasets across reports to reduce duplicate calls.
- Keep credentials and generated report data session-only by default.
- Provide placeholder credential-acquisition guidance that can be filled in later.

## Non-Goals For The MVP

- No persisted credentials.
- No persisted report data in `localStorage` or `IndexedDB`.
- No backend/proxy/runner service.
- No SCIM activation, deactivation, or deletion execution.
- No API import, user group mutation, or other write operations.
- No Selenium-driven admin-page scraping in the browser app.
- No full cross-report analytics platform beyond a light session overview.

## Browser-Ready Report Catalog

### Tag Report

Source: `so4t_tag_report`

Live path: Stack API v2.3 and v3 reads for tags, users, questions, articles, tag SMEs, and related content.

Upload path: existing `tag_metrics.csv` and generated JSON data where available.

Dashboard ideas:

- top tags by page views, watchers, question count, answer count, SME count
- median time to first answer and first response
- unanswered and accepted-answer rates
- webhook/community counts where available from imported data or future supported collectors
- raw tag metrics table

The optional Python `--web-client` path depends on Selenium/admin-page scraping and is not part of browser execution.

### API User Report

Source: `so4t_api_user_report`

Live path: Stack API v2.3 and v3 reads for users, reputation history, questions, articles, tags, communities, and related activity.

Upload path: existing `user_metrics.csv` and JSON data files.

Dashboard ideas:

- top contributors by net reputation, answers, questions, articles, comments
- account status distribution
- inactivity distribution
- SME tag coverage
- department/title breakdown where available
- raw user metrics table

The commented or optional web-client scraping path is not part of browser execution.

### Inactive Users

Source: `so4t_inactive_users`

Live path: Stack API v2.3 reads for users with activity and contribution fields.

Upload path: existing inactive-user CSV outputs.

Dashboard ideas:

- inactive users by days inactive
- contributing vs non-contributing inactive users
- deactivated vs registered split
- reputation/content-risk distribution
- raw inactive-user table

### Interactions Report

Source: `so4t_interactions_report`

Live path: Stack API v2.3 and v3 reads for users and questions/answers/comments. The Python implementation uses pandas and d3blocks; the browser app should port the transformation to TypeScript and render the network/chord-style visualization with browser-native charting.

Upload path: generated `interaction_data.json`, `interaction_matrix.csv`, or compatible CSV/JSON exports where available.

Dashboard ideas:

- department-to-department interaction matrix
- chord/network visualization
- top source and target departments
- untracked/deleted-user warning counts
- raw interaction table

### Community Members

Source: `so4t_community_members`

Live path: Stack API v3 reads for communities, community details, and users for email/job-title/department lookup.

Upload path: existing timestamped community members CSV files.

Dashboard ideas:

- member counts by community
- SME distribution
- job title and department distribution
- raw community member table

### Data Export

Source: `so4t_data_export`

Live path: Stack API v2.3 and v3 reads for users, user groups, tags, articles, questions, answers, and comments.

Upload path: existing JSON exports for users, user groups, tags, articles, and questions/answers/comments.

Dashboard ideas:

- export completeness summary
- dataset counts and timestamps
- dataset-specific raw tables
- JSON download bundle

## Excluded Or Later-Phase Scripts

### WebHook Report

Source: `so4t_WebHook_report`

Reason excluded: depends on Selenium, a logged-in browser session, admin page access, and HTML scraping of Enterprise admin pages. This is not cleanly executable from a static browser app using API credentials alone.

Later phase: catalog entry with external runbook, or a customer-approved local runner/backend if needed.

### Search Log Report

Source: `so4t_search_log_report`

Reason excluded: depends on Selenium login flow and scraping `/developer/logs/3` pages. Not a browser-only static API workflow.

Later phase: catalog entry with external runbook, or a customer-approved local runner/backend if needed.

### SCIM User Activation, Deactivation, And Deletion

Sources:

- `so4t_scim_user_activation`
- `so4t_scim_user_deactivation`
- `so4t_scim_user_deletion`

Reason excluded: these use SCIM endpoints and mutation methods such as `PATCH`, `PUT`, and `DELETE`. They are privileged operational actions, not read-only reporting. CORS behavior may vary by customer instance, and destructive operations need a much more guarded workflow.

Later phase: separate guarded administration area, likely requiring explicit dry runs, confirmation steps, and possibly a runner/backend.

### API Import And User Groups

Sources:

- `so4t_api_import`
- `so4t_user_groups`

Reason excluded from MVP: both perform write operations. `so4t_api_import` creates questions, answers, or articles. `so4t_user_groups` creates groups or adds members. They may be technically feasible through browser API calls, but they should ship in a later guarded phase after read-only reporting is stable.

## Product Decisions

- MVP direction: Reporting MVP first, then phased expansion.
- Data sources: both live API calls and uploaded existing CSV/JSON outputs.
- Credential retention: session-only.
- Report data retention: session-only.
- Report catalog: all browser-ready read-only reports.
- UI foundation: Stacks design system.
- Instance coverage: both Basic/Business and Enterprise, with explicit report availability.
- Workspace layout: Report Catalog.
- Credential UX: shared Credentials screen plus report-specific scope notes.
- Dashboards: per-report dashboards plus a light cross-report session overview.
- Data inspection: raw table with filtering/search/sort/export for every report.
- Run model: single report runs plus selected-report batch queue.
- Delivery: static web app.
- Implementation approach: native TypeScript port.

## Architecture

The app is a static TypeScript frontend. It can be hosted by any static file host and does not require an application server for the MVP.

Recommended high-level units:

- `app shell`: top navigation, report catalog layout, session overview, route/view state
- `credentials`: session-only credential state, instance-type selection, validation, placeholder guidance
- `api clients`: Stack API v2.3 and v3 clients, URL normalization, pagination, backoff, error mapping
- `report registry`: report metadata, availability rules, required credential/scope notes, run/import capabilities
- `collectors`: live API data collection per shared dataset
- `importers`: CSV/JSON parsers for existing script outputs
- `transformers`: pure TypeScript functions that compute report rows and metrics
- `workers`: Web Worker execution for collection, parsing, and transformation
- `tables`: reusable data grid behavior for search, filtering, sorting, visibility, and export
- `dashboards`: reusable chart/card components plus report-specific dashboard layouts
- `exports`: CSV/JSON download generation

Report modules should communicate through stable internal dataset types rather than through UI-specific structures. This keeps live API results and uploaded script outputs interchangeable once normalized.

## Data Flow

### Live API Collection

1. User opens the shared Credentials screen.
2. User selects Basic/Business or Enterprise and enters the required URL and session-only credentials.
3. User chooses one report or selects multiple reports for batch execution.
4. The run planner determines required shared datasets.
5. Web Workers call the shared API clients and collect pages.
6. Backoff/throttle waits are shown in run status.
7. Raw datasets are normalized into session memory.
8. Report transformers compute report rows and dashboard metrics.
9. Dashboards and tables render in the selected report pages and session overview.
10. User can explicitly download CSV/JSON outputs.

### Uploaded Output Import

1. User opens Uploads or a report page upload control.
2. User drops CSV/JSON files from current Python script outputs.
3. Importers validate file shape, columns, and report compatibility.
4. Parsed data is normalized into the same internal structures used by live API runs where possible.
5. Compatible dashboards and raw tables become available without credentials.
6. Validation warnings stay visible and report-scoped.

## Error Handling

- One failed report does not clear loaded datasets or block unrelated report views.
- Live API errors are surfaced with endpoint, status, and report/dataset context.
- CORS failures are treated as environment limitations and explained without exposing credentials.
- Missing credential/scope errors point to the shared Credentials screen and report-specific scope notes.
- Upload validation errors list missing columns, unsupported file types, and expected output sources.
- Backoff and rate-limit waits appear in the run queue instead of looking like a frozen app.

## UI Design

The app opens directly to the Report Catalog workspace.

Top-level navigation:

- app name
- Credentials
- Uploads
- Help

Main layout:

- left-side report catalog with the six MVP reports
- selected report workspace
- optional session overview when data is loaded

Each report page includes:

- report name and source-script context
- availability badge for Basic/Business, Enterprise, or both
- credential/scope notes with placeholders
- live API run controls
- upload existing output controls
- run status and warnings
- dashboard view
- raw table view
- export controls

Session Overview includes:

- loaded datasets
- report freshness
- total users/tags/content where known
- warnings
- quick links to report dashboards

The interface should be dense, practical, and operational. It should avoid marketing-page composition and use Stacks-style forms, notices, tables, tabs, and controls.

## Security And Privacy

- Credentials stay only in JavaScript memory.
- Generated report data stays only in JavaScript memory.
- No credential or report data persistence in `localStorage`, `sessionStorage`, or `IndexedDB` in the MVP.
- Downloads are explicit user actions.
- Uploads are parsed locally in the browser.
- The app does not send data to any service except the user-entered Stack instance/API endpoints during live runs.
- No write/mutation capabilities are included in the MVP.

## Testing Strategy

Use tests around pure modules and browser flows.

Core module tests:

- Basic/Business URL normalization
- Enterprise URL normalization
- credential requirement validation
- report availability rules by instance type
- API pagination
- v2.3 backoff handling
- v3 throttle-header handling
- API error mapping
- CSV parsing for sample outputs
- JSON parsing for sample outputs
- report transformer outputs against fixture data
- export generation

UI/browser tests:

- shared Credentials screen accepts session credentials without persistence
- selected report live-run setup displays required scopes
- upload-only mode enables dashboards without credentials
- single report run status progresses and completes
- batch queue reuses shared datasets
- report failure leaves other loaded data available
- raw tables search, filter, sort, and export
- dashboards render at desktop and mobile widths
- session overview appears only after data is loaded

## Phased Expansion

Phase 1: Reporting MVP described in this spec.

Phase 2: richer dashboards, saved layout preferences without customer data, and more cross-report analytics.

Phase 3: guarded write tools such as API Import and User Groups, with dry-run previews, diff tables, explicit confirmations, and audit-friendly output.

Phase 4: non-browser-only tools such as SCIM operations, WebHook Report, and Search Log Report via external runbooks or an approved local runner/backend.
